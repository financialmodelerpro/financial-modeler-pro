No migration. **Phase M1.8 fix 5 (2026-05-04, commit 66a20f5)**, every snapshot the system POSTs (wizard create, legacy create, auto-save) is bare `HydrateSnapshot` with **no `version: 3` discriminator field at the top level**, so the recogniser at `lib/state/module1-migrate.ts:isNewV3` was falling through to the legacy v2 path on reload and replaying default-seed (`projectName: 'Skyline'`, `assets:0`, `plots:0`). `isNewV3` is now SHAPE-BASED: `(version === 3 || version === undefined) && Array.isArray(assets) && Array.isArray(phases) && Array.isArray(costs)`. Older snapshots that explicitly carry `version: 3` still match; new bare snapshots also match. Fix is read-side only, no DB migration required, no schema change to the JSONB shape, existing rows untouched. **Migration 152 (2026-05-31)** adds `base_version_id uuid` + `change_log jsonb DEFAULT '[]'` (session-based versioning). **Migration 153 (2026-06-01)** adds nullable `version_label` / `task_name` / `comment` (auto version naming `{Project}_v{X.Y}_{MMDDYYYY}_{Task}` stored in `label`, plus a required per-version comment). Both additive + idempotent (`ADD COLUMN IF NOT EXISTS`) and server-tolerant (`server.ts` try-FULL-fall-back-to-BASE behind the `m152Applied` probe, synthesising null defaults). Both must be applied manually to production Supabase per the project convention; until then the new fields read as null and the platform keeps working. |
